The design choices behind looped transformers

Most looped language models share the same broad structure: prepare the representation, run a repeated block, decode the result. The differences matter: what is shared, where recurrence happens, how the model exits, and how stability or capacity is recovered.

Shared pattern
prelude / loop / coda

Modern designs keep input handling and output decoding outside the repeated core.

Capacity move
hyper-connections

Hyperloop expands what the repeated middle can carry between passes.

Depth move
fixed, sampled, learned

R can be a constant, a training distribution, or a token-level exit decision.

01

The common skeleton

Pattern
shared shape
A looped model usually has a front block, a repeated middle block, and an output block. The repeated block is where extra test-time compute enters.

Huginn makes this structure clear: prelude, recurrent block, coda. Hyperloop uses begin, middle, and end blocks. Parcae also fits this mental model through its input injection and stable recurrent block. Ouro is organized around repeated steps with a learned exit gate.

The shared middle block is the parameter-saving move. A model can run the same block several times, spending compute without storing a separate set of parameters for every realized layer.

front
Prepare the representation.

The first layers handle token-level features and put the prompt into the working residual space.

loop
Spend repeated compute.

The middle block is reused, so realized depth grows faster than stored parameter count.

exit
Decode from the final state.

The coda and unembedding see a residual state that has passed through the recurrent computation.

Four things to track

For each design, separate the stored parameters from the realized computation. Then look at the residual state that crosses loop boundaries, the rule that chooses R, and the mechanism that keeps the repeated map stable.

03

Hyperloop’s specific move

Hyper-connections
Hyperloop
Hyperloop keeps middle-block reuse and adds matrix-valued residual streams at loop boundaries to recover capacity.

Plain looping saves parameters, but it also forces repeated computation through a narrow residual interface. Hyperloop adds hyper-connections after loop iterations. These create matrix-valued residual streams, giving the looped middle block more room to carry distinct state across repetitions.

The paper reports that Hyperloop outperforms depth-matched transformer and mHC baselines while using about 50% fewer parameters. It also studies INT4 post-training quantization, because parameter efficiency is less useful if the model becomes hard to deploy at low precision.

Where the parameter saving comes from

The middle block is reused. The realized depth is larger than the stored depth. Hyper-connections add capacity at loop boundaries without replacing the parameter-sharing idea.

Vanilla loop

Each pass writes back into the same residual stream. This is parameter-efficient, but repeated state can become cramped.

Hyperloop

Hyper-connections mix multiple residual streams after loop iterations, giving the repeated middle a wider state interface.

04

Parcae solves a different problem

Parcae
Parcae is mainly about training dynamics: keep recurrence stable so loops can scale predictably.

Hyperloop focuses on parameter-efficient architecture. Parcae focuses on stable training. Its negative diagonal parameterization and per-sequence sampling are there to prevent the loop from learning unstable recurrence dynamics.

These ideas can be complementary. A practical looped model needs both enough capacity inside the repeated computation and stable dynamics across recurrence.

05

Ouro changes the depth decision

Ouro
Ouro moves recurrence depth from a global setting into a learned per-token exit distribution.

Ouro's main architectural addition is the exit gate. Rather than choosing one fixed recurrence count for every token, the model can assign a probability of exiting at each loop step.

That makes the loop budget a learned decision. The architecture and objective are tied together: a language modeling loss at each step, an exit probability at each step, and an entropy-regularized objective to keep the exit distribution usable during training.

06

A practical design checklist

design review
A looped design should be judged by recurrence placement, input preservation, state capacity, depth choice, and stability control.
  • Where is recurrence placed? Full stack, middle block, or a specialized stable block.
  • How is the input preserved? Input injection and normalization choices decide whether the prompt remains available across loops.
  • How is capacity preserved? Hyper-connections, larger recurrent blocks, or learned exits can all change what the loop can carry.
  • How is depth selected? Fixed R is simple. Sampled R trains scalability. Learned exits make compute adaptive.
  • How is instability controlled? Spectral constraints and recurrence-aware training matter once loop depth grows.
