How a looped model decides how long to think

Test-time compute in these papers is recurrence depth. Huginn shows that extra recurrence can improve reasoning. Ouro adds a learned exit mechanism so the model can stop earlier on easier inputs.

01

Huginn: spend more loops at inference

Recurrent depth
paper result
Huginn trains a recurrent-depth language model and evaluates it at larger recurrence counts during inference.

The model has a prelude, a shared recurrent block, and a coda. The recurrent block can be run multiple times before the final output. In the paper, harder reasoning tasks benefit from extra recurrence more than simpler knowledge tasks.

The reported run is a 3.5B parameter model trained on 800B tokens. The authors evaluate recurrence counts up to 64 and show gains on reasoning-heavy evaluations such as GSM8K-style tasks.

What the curve means

A test-time compute curve usually rises quickly, then flattens. Early loops fix the obvious parts of the representation. Later loops help only if the task still has unresolved computation.

02

Zero-shot stopping with state change

Huginn exit
A threshold on successive-state change can allocate recurrence without training a separate exit head.

Huginn also explores adaptive exits without training a separate exit head. The idea is to compare successive recurrent states. If the state changes very little, the model can stop. The paper uses KL-style differences between steps for this kind of thresholding.

This is attractive because it does not require labels for “how many loops should this token get?” It uses the model's own convergence behavior as the signal.

03

Ouro: learn the exit

Ponder gate
Ouro exit
Ouro makes depth allocation part of training by predicting an exit probability at each recurrent step.

Ouro makes the exit decision explicit. At each recurrent step, an exit gate predicts a probability of stopping. The language-modeling head also produces a loss at each step, so training can connect loop depth with prediction quality.

A learned gate can collapse if the objective is poorly shaped. It may stop too early, or it may always use the deepest step. Ouro uses an entropy-regularized objective with a prior over exit steps, then trains the gate with a specialized adaptive-exit loss.

Entropy regularization

Keeps the exit distribution from collapsing during pretraining.

Adaptive exit training

Trains the gate to stop when another recurrent step is unlikely to pay off.

04

Compute appetite differs by task

task behavior
Reasoning-heavy tasks keep using recurrence longer. Knowledge-heavy tasks usually flatten earlier.

A math problem and a fact recall question do not use recurrence the same way. In the papers, reasoning-heavy tasks tend to keep improving for more recurrence steps. Tasks that depend more on stored knowledge saturate earlier.

The useful deployment question is per input, not per benchmark: how many loops does this token need before the state is good enough to decode?

Reasoning-heavy input

The residual keeps changing for longer because the model is still composing intermediate quantities.

Knowledge-heavy input

The relevant fact is usually present early. Later loops add little once the state points at the answer.

05

Memory during decoding

deployment detail
Adaptive compute has a KV-cache cost. Ouro tests cache reuse policies because loop depth changes decoding memory.

Ouro also studies KV-cache sharing. A 4-step recurrent model can require separate caches for each recurrent step. During decoding, the paper finds that last-step cache reuse and averaged reuse preserve performance much better than first-step reuse.

This matters because adaptive compute only helps deployment if the memory story holds. Looping can save parameters, but repeated states still have runtime costs.
